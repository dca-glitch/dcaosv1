## Summary rules confirmed by code inspection

- Every integration follows the same **disabled → missing_config → configured_shape_ok** three-state pattern, and every readiness category explicitly self-labels as live-calls-deferred (`liveCallsDeferred: true` / equivalents) at the type level in `external-integrations-readiness.service.ts`.
- No integration in this codebase currently claims live-proven status anywhere in code or docs that this audit could find — all live proof requirements are consistently deferred pending separate owner approval, consistent with `G53_PRODUCTION_SAFETY_PLAN.md` §4.2.
- Two gaps for a future small block (not blockers, just completeness): (1) image generation has scaffold but no readiness category yet — add when provider is wired per `IMAGE_GENERATION_PROOF.md`; (2) email/Resend readiness is not yet exposed via the unified `GET /api/v1/integrations/readiness` endpoint — it is only visible via the email outbox smoke.
- **Email notifications audit (2026-07-09):** Full pathway/event audit completed in [`EMAIL_NOTIFICATIONS_PROOF.md`](./EMAIL_NOTIFICATIONS_PROOF.md). Headline finding: of the six Puriva-launch notification events (article ready, image set ready, client approved, client rejected, admin action required, monthly report final, WordPress draft prepared), only client-approved and client-rejected currently reach a real inbox; the rest are either internal-only (`SKIPPED`-only placeholder log) or entirely unwired. No live email was sent to produce that audit.
- **SEC-B1 (2026-07-09):** Legacy `/api/v1/briefs` cross-tenant IDOR fixed locally; regression test `briefs-tenant-boundary.integration.test.ts`. Not a launch integration row but a production safety prerequisite.
- **WordPress live draft proof (2026-07-09):** `WORDPRESS_DRAFT_PROOF.md` §6 documents a pre-execution plan only — no live WordPress call was made to produce it or this matrix update. Three concrete code/schema gaps must be explicitly decided (schema change vs. manual-check-only) before a live session can run: no `altText`/`caption`/`socialPreview` fields on `AiDeliveryArticleImage`, no idempotency/dedupe key on `PublicationLog`/`AiDeliveryWordPressPublishRequest`, and no code-level filter enforcing that only `AiDeliveryDeliverableImageApproval.status = APPROVED` images may attach to a draft.
- **GA/GSC token readiness (2026-07-09, docs-only):** Confirmed by code inspection that GA/GSC readiness is env-presence-only — no OAuth consent/callback route, token storage model, or refresh logic exists in `apps/api` or `packages/data`. This is a harder blocker than "complete OAuth consent screen" alone implies; a token storage/encryption design block must be scoped and approved before any consent flow is attempted. See `MONTHLY_REPORT_LIVE_DATA_PROOF.md` §3.1a.
